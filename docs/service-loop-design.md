# 录音数据服务闭环设计

> T+1 模式：昨天发生了什么 → 今天能为你做什么

---

## 整体架构

```
录音拉取(23:00) → 感知层(23:40) → 理解层(23:45) → 服务生成层(23:50) → 推送层(次日8:30)
                                                        ↑
                                                  用户画像(持续更新)
                                                  历史意图/事件库
```

新增核心模块：**服务生成层**（`src/services/`），负责把理解转化为具体服务。

---

## 服务 1：晨间 Brief

### 是什么
每天早上一条飞书消息，不是流水账，而是"今天你可能需要关注什么"。

### 示例输出
> ☀️ 早上好
>
> **昨天要点**
> • 和超哥讨论了记忆功能的 API 重构，他倾向于用 streaming 方案，你更想先跑通 batch —— 这个分歧还没收口
> • 下午跟设备组联调了 2 小时，版本回滚了一次，晓龙说硬件端周四才能给新固件
> • 晚上健身了（背+二头），状态不错
>
> **今天建议关注**
> • API 方案跟超哥对齐一下，拖久了会卡后面的排期
> • 晓龙的固件明天到，可以提前准备测试环境

### 数据来源
- `macro_frames` → key_topics, primary_activity, mood_or_tone, outcomes
- `scenes[activity=meeting]` → svo_bullets 中的未决项
- `intents[type=todo, status=active]` → 最近新增的待办

### 处理逻辑
1. 从昨天录音的 macro_frames 提炼 top 3-5 事件（按 coverage.ratio 排序）
2. LLM（Haiku）生成简报，prompt 含用户画像 + 昨天情绪
3. 情绪自适应：mood=stressed → 语气柔和，mood=energetic → 语气轻松
4. 从 intents 中找 "今天需要跟进"的项

### 触发
- cron 每天 8:30（工作日）
- 数据依赖：昨晚 23:50 服务生成层产出 `memory/services/daily_brief_YYYYMMDD.json`

### 输出
- 飞书消息（L2 推送）
- 同步到 Prism 便签模式（18-20点显示）

## 服务 2：会议洞察

### 是什么
不是会议纪要（录音转写已经有了），而是你在会议中的"元认知"——你的角色、分歧、悬而未决的问题。

### 示例输出
> 📋 昨天 3 场会议洞察
>
> **10:00 晨会**（6人，35min）
> • 你主导了议程，发言占比约 40%
> • 关键决策：决定先跑 batch 方案做 POC
> • ⚠️ 悬而未决：数据上传的鉴权方式还没定，需要找欧金梁确认
>
> **14:00 设备联调**（3人，2h）
> • 你主要在倾听和调试（发言占比 15%）
> • 版本回滚了一次，根因是固件版本不匹配
> • 行动项：等晓龙周四给新固件
>
> **17:30 1on1 超哥**（25min）
> • 讨论了排期压力，超哥建议先砍掉非核心功能
> • 你没有明确表态 —— 要不要今天想清楚回复他？

### 数据来源
- `scenes[activity=meeting]` → participants, svo_bullets, key_quotes, summary
- `entity_canon.people` → 参会者身份
- `macro_frames` → 该时段的 mood_or_tone

### 处理逻辑
1. 过滤 meeting 类 scenes，按时间分组
2. 计算用户发言比例（从 key_quotes 中 speaker=用户 的占比）
3. LLM（Haiku）分析每场会议的：决策点、分歧、未决项、行动项
4. 特别标注"你没有回应但对方在等"的信号

### 触发
- 随晨间 Brief 一起生成
- 只在有 meeting 场景时触发（没开会就不推）

### 输出
- 飞书消息（跟晨间 Brief 合并或单独发，看长度）

## 服务 3：意图追踪与行动

### 是什么
你说的每一句"想做XX"、"要办XX"，不再只是存一条记录，而是 T+1 就有动作。

### 意图分类与响应

| 类型 | 示例 | 自动响应 |
|------|------|---------|
| **wish**（想要） | "想去福州玩" "想买个降噪耳机" | T+1 搜索信息→推方案 |
| **todo**（要做） | "明天要给XX发邮件" "进京证要办了" | 创建跟踪→设提醒→到期催 |
| **idea**（灵感） | "这个可以做成skill" "小红书可以试试这个方向" | 存灵感库→关联历史想法→周回顾 |
| **plan**（计划） | "下周把API改成streaming" | 进入 todo.md→Daily Brief 提及 |

### 示例输出（wish 类）
> 💡 你昨天提到想去福州玩，我查了一下：
> • 直飞：北京→福州 2.5h，最近票价 ¥450-680（南航/厦航）
> • 天气：下周福州 18-25°C，晴为主
> • 推荐季节：3-5月最舒适
> 要我出个详细行程方案吗？

### 示例输出（todo 类，跟踪提醒）
> ⏰ 你 8 天前提到要办皮肤科挂号（海淀医院），还没动。
> 每天早 7:00 放号，要我明早提醒你？

### 数据来源
- `scenes[].key_quotes` → 意图原文
- `intents.json` → 已分类的意图库
- LLM 分类新意图（从当天录音的 key_quotes 中提取）

### 处理逻辑
1. 每日 pipeline 中，从新录音提取 key_quotes
2. LLM（Haiku）分类：wish/todo/idea/plan/noise
3. wish → 调用 master-search / flightclaw 搜索，生成推送内容
4. todo → 写入 intents.json(status=tracking)，设 remind_at
5. idea → 写入 idea-capture.md，LLM 检查是否有历史类似想法
6. plan → 写入 todo.md

### 意图生命周期
```
new(录音提取) → classified(LLM分类) → tracking(跟踪中) → actioned(已推送方案/已提醒) → done/expired
```

### 周度回顾（每周日）
> 📊 本周意图回顾
> • ✅ 完成 3 项（办了XX、买了XX、查了XX）
> • 🔄 进行中 5 项
> • ❓ 拖了 2 周+：XXX —— 还要继续跟吗？

### 触发
- 新意图：daily_pipeline 中自动提取
- wish 响应：T+1 晨间推送
- todo 提醒：按 remind_at 定时
- 周回顾：每周日晚

## 服务 4：人际洞察（周度）

### 是什么
基于录音中的社交数据，每周告诉你人际网络的变化和建议。不是冷冰冰的统计，而是有温度的观察。

### 示例输出
> 🤝 本周人际动态
>
> **高频互动**：超哥（14次）、晓龙（8次）、欣钰（5次）
> **明显升温**：和晓龙的互动比上周多了2倍，说明联调进入深水区
> **值得注意**：和宁深哥已经 2 周没聊了——他上次提到在准备健身比赛，要不要周末约一场？
> **新面孔**：本周出现了"王彤"，看起来是新入职的同事

### 数据来源
- `relationships.json` → 关系强度、互动频率、趋势
- `entities.json[people]` → 人物信息
- `events.jsonl` → 本周 interaction 类事件

### 处理逻辑
1. 统计本周 vs 上周的人物互动频率变化
2. 检测：新出现的人、消失的人、频率突变的人
3. LLM 生成自然语言观察（融入上下文，不是干巴巴的数据）
4. 提炼 1-2 条可行动的建议

### 触发
- weekly_refine.py 周日跑完后生成
- 周一早 8:30 随晨间 Brief 推送

## 服务 5：情绪关怀

### 是什么
检测到高压信号时，自然地表达关心。不说"系统检测到你情绪低落"，而是像朋友一样。

### 触发信号
| 信号 | 数据来源 | 权重 |
|------|---------|------|
| 连续 2 天 mood=stressed/frustrated | macro_frames.mood_or_tone | 高 |
| 当天加班到 22:00+ | scenes 的 end_time | 中 |
| 会议中出现争论（key_quotes 有对立语义） | svo_bullets | 中 |
| 连续 5 天无运动 | Habit Predictor 数据 | 中 |
| 一整天无社交录音（独处异常长） | scenes 统计 | 低 |

### 示例输出（加班+高压）
> 今天强度不小啊，联调+开会塞满了一整天。
> 距离上次去亚C也有 5 天了，要不明晚安排一次？周三练背怎么样？

### 示例输出（争论后）
> 下午那场讨论确实有点僵，不过你的方案思路是对的，数据支撑也够。
> 有些事不急着一次说服，可以让结果说话。

### 处理逻辑
1. daily_pipeline 结束后，检查情绪信号
2. 多信号叠加时触发（单一信号不触发，避免误报）
3. LLM 生成关怀消息：知道发生了什么 → 共情 → 轻建议
4. 融合运动/社交数据给具体建议

### 输出
- 飞书消息（L2，但不在安静时间推）
- Prism 屏幕 NOTE（低侵入）

## 服务 6：兴趣与偏好学习

### 是什么
持续从录音中提取兴趣信号，更新用户画像，让其他服务更精准。

### 提取维度
| 维度 | 信号来源 | 示例 |
|------|---------|------|
| 话题兴趣 | key_topics 频率统计 | "最近经常讨论 MCP 协议" |
| 产品/品牌 | key_quotes 中的实体 | "提到了 3 次 Claude" |
| 人/内容偏好 | 录音中提到的内容创作者 | "聊到了林亦的视频" |
| 消费倾向 | 价格/购买相关 key_quotes | "觉得这个耳机性价比不错" |
| 情绪锚点 | 什么场景/话题让你开心 | "聊健身时 mood=energetic" |

### 处理逻辑
1. 每日 pipeline 中，从 macro_frames.key_topics 统计频率
2. 每周 refine 时，LLM 分析兴趣趋势变化
3. 更新到 profile.json 的 preferences 字段
4. 其他服务引用：晨间 Brief、周末推荐、新闻推送都基于兴趣画像

### 输出
- 不直接推送（被动积累）
- 通过其他服务体现："你最近关注 MCP 比较多，这有篇不错的实践文章"

## 服务 7：模式异常检测

### 是什么
发现日常模式偏离时，自然地关心，不报警式通知。

### 检测项
| 异常 | 检测方法 | 响应 |
|------|---------|------|
| 比平时晚到 2h+ | 录音首条 scene 的 start_time vs 历史中位数 | "今天到得晚，还好吗？别急" |
| 一整天无社交 | scenes 中无 interaction 类型 | 不推送，但记录 |
| 运动频率突降（2周+未运动） | Habit Predictor | 轻推建议 |
| 加班频率突增（连续 3 天 22:00+） | scenes end_time 统计 | "这周节奏挺紧的" |
| 饮食异常（无 dining 场景） | scenes[activity=dining] 缺失 | "忙到没吃饭？" |

### 处理逻辑
1. 每日 pipeline 比较当天模式 vs 历史基线（profile.json + patterns.json）
2. 偏离超过阈值→标记异常
3. 多天偏离叠加才触发推送（单日偏离正常）
4. LLM 生成自然语气消息

## 服务 8：上下文记忆增强

### 是什么
和你聊天时，能引用录音中的具体场景，让你感觉"被记住了"。

### 示例
饭团说："超哥那个方案怎么样了？"
> 你说的是 3/12 下午跟超哥讨论的 API streaming 方案吧？当时他倾向 streaming，你更想先跑 batch POC。后来 1on1 的时候他又建议先砍非核心功能。你最后没有明确回复他——要不要今天对齐一下？

### 实现方式
1. 不是新模块，而是增强主 session 的上下文能力
2. MEMORY.md 和 daily memory 中融入录音关键事件
3. 对话中被问到某人/某事时，检索 events.jsonl + intents.json

### 触发
- 对话中自然触发（不用 cron）
- 通过 local-embedding 做语义搜索匹配

## 关键设计原则

1. **不做转述，做洞察** —— 不是"你昨天开了3个会"，而是"超哥那个分歧还没收口"
2. **不做通知，做服务** —— 不是"检测到意图"，而是直接查好了推给你
3. **语气像朋友** —— 知道你怎么说话、什么时候需要推一把、什么时候该安静
4. **越用越准** —— 兴趣画像、情绪基线、人际图谱都是持续更新的
5. **宁缺毋滥** —— 没有有价值内容时不推送，不凑数
